# SignPost Design Spec

**Date:** 2026-03-29
**Status:** Draft
**Author:** drose + Claude

## Context

Local services (monitoring, web apps, cron jobs) need to send email notifications, but modern email authentication requirements (DKIM, SPF, DMARC) cause these emails to be rejected. SignPost solves this by providing a local SMTP server that properly DKIM-signs outgoing mail and relays it through a configurable smarthost (Gmail, ISP, or direct delivery). It includes a web admin UI that guides users through correct DNS configuration and provides built-in testing tools.

The project targets two deployment environments:
- **Dev:** Local WSL Ubuntu with Docker
- **Prod:** ubuntu01.drcs.ca (root access, Cloudflare DNS for drcs.ca)

## Architecture

### Single Container, Two Processes

SignPost runs as a single Docker container with two processes managed by s6-overlay:
- **Maddy** (foxcpp/maddy v0.9.x) — all-in-one mail server handling SMTP, DKIM signing, SPF/DMARC, TLS/STARTTLS
- **SignPost Web** — Go backend + React frontend for configuration, monitoring, and testing

```
┌──────────────────────────────────────────────┐
│              SignPost Container                │
│                                                │
│  ┌─────────────┐    ┌──────────────────────┐  │
│  │   Maddy      │    │   SignPost Web (Go)   │  │
│  │   :25 SMTP   │    │   :8080 HTTP          │  │
│  │   :587 Sub   │    │   REST API + React SPA│  │
│  └──────┬───────┘    └──────────┬───────────┘  │
│         │                       │              │
│         ▼                       ▼              │
│  ┌─────────────────────────────────────────┐  │
│  │         /data/signpost/                  │  │
│  │  ├── maddy.conf (generated by web app)   │  │
│  │  ├── signpost.db (SQLite)                │  │
│  │  ├── dkim_keys/ (auto-generated)         │  │
│  │  ├── tls/ (LE or manual certs)           │  │
│  │  └── backups/ (encrypted archives)       │  │
│  └─────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

### Data Flow

```
Local Service → SMTP :25 (network trust) or :587 (SMTP AUTH + STARTTLS)
    → Maddy receives message
    → Maddy signs with DKIM (per-domain key)
    → Maddy relays via configured smarthost (Gmail/ISP/direct)
    → Recipient mail server verifies DKIM/SPF/DMARC
    → Email delivered
```

### Config Generation

The web app owns the configuration. User changes in the UI are:
1. Saved to SQLite
2. Rendered into `maddy.conf` via Go templates
3. Maddy is signaled to reload (SIGHUP)

This means Maddy's config file is never manually edited — it's always generated.

## Authentication Model

- **Port 25 (SMTP):** Network trust by default — any service on the Docker network or localhost can send without credentials. Configurable to require auth.
- **Port 587 (Submission):** SMTP AUTH required over STARTTLS. Username/password managed in web UI.
- **Web UI (:8080):** Built-in basic auth. Designed to sit behind a reverse proxy (nginx proxy manager) for production. Optional standalone HTTPS with LE certs for users without a reverse proxy.

## TLS Strategy

- **Inbound SMTP (local services):** Self-signed by default, configurable to use LE or manual certs
- **Outbound relay:** STARTTLS to relay server (enforced by default, configurable)
- **Web UI:** HTTP by default (sits behind reverse proxy), optional LE cert generation for standalone deployments
- **Let's Encrypt method:** ACME DNS-01 challenge via Cloudflare API (works behind NAT, no port 80 needed)

## Database Schema

SQLite with migration versioning (golang-migrate or custom).

```sql
-- Core domain configuration
CREATE TABLE domains (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT NOT NULL UNIQUE,          -- e.g. "drcs.ca"
    dkim_selector TEXT NOT NULL DEFAULT 'signpost', -- e.g. "signpost"
    dkim_key_path TEXT,                        -- path to private key file
    dkim_public_dns TEXT,                      -- the DNS TXT record value
    dkim_created_at DATETIME,
    spf_record  TEXT,                          -- recommended SPF record
    dmarc_record TEXT,                         -- recommended DMARC record
    active      BOOLEAN NOT NULL DEFAULT 1,
    created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- Per-domain relay configuration
CREATE TABLE relay_configs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    domain_id   INTEGER NOT NULL REFERENCES domains(id) ON DELETE CASCADE,
    method      TEXT NOT NULL DEFAULT 'direct', -- 'gmail', 'isp', 'direct', 'custom'
    host        TEXT,                           -- relay hostname e.g. "smtp.gmail.com"
    port        INTEGER DEFAULT 587,
    username    TEXT,
    password_enc TEXT,                          -- AES-256-GCM encrypted
    password_nonce TEXT,                        -- nonce for AES-GCM decryption
    starttls    BOOLEAN NOT NULL DEFAULT 1,
    created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(domain_id)
);

-- SMTP auth users (for port 587 submission)
CREATE TABLE smtp_users (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    username    TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,                -- bcrypt hash
    active      BOOLEAN NOT NULL DEFAULT 1,
    created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- Which domains an SMTP user can send as
CREATE TABLE smtp_user_domains (
    smtp_user_id INTEGER NOT NULL REFERENCES smtp_users(id) ON DELETE CASCADE,
    domain_id    INTEGER NOT NULL REFERENCES domains(id) ON DELETE CASCADE,
    PRIMARY KEY (smtp_user_id, domain_id)
);

-- TLS configuration
CREATE TABLE tls_config (
    id          INTEGER PRIMARY KEY CHECK (id = 1), -- singleton row
    mode        TEXT NOT NULL DEFAULT 'self-signed', -- 'acme', 'manual', 'self-signed'
    acme_email  TEXT,
    acme_provider TEXT DEFAULT 'cloudflare',
    cert_path   TEXT,
    key_path    TEXT,
    cert_expiry DATETIME,
    updated_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- Global settings
CREATE TABLE settings (
    key         TEXT PRIMARY KEY,
    value       TEXT NOT NULL
);
-- Default settings rows:
-- 'network_trust_enabled' = 'true'
-- 'network_trust_cidrs' = '172.16.0.0/12,127.0.0.1/8'  (Docker + localhost)
-- 'smtp_port' = '25'
-- 'submission_port' = '587'
-- 'web_port' = '8080'
-- 'backup_encryption_key_hash' = '' (bcrypt hash of user-provided backup password)

-- Mail log (recent activity, not long-term storage)
CREATE TABLE mail_log (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    from_addr   TEXT NOT NULL,
    to_addr     TEXT NOT NULL,
    domain_id   INTEGER REFERENCES domains(id),
    subject     TEXT,
    status      TEXT NOT NULL,                 -- 'sent', 'failed', 'deferred'
    relay_host  TEXT,
    error       TEXT,
    dkim_signed BOOLEAN DEFAULT 0
);

-- Schema version tracking
CREATE TABLE schema_migrations (
    version     INTEGER PRIMARY KEY,
    applied_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

## Network Trust Definition

"Network trust" on port 25 means Maddy accepts mail without SMTP AUTH from IP addresses matching configured CIDR ranges. Default CIDRs:
- `172.16.0.0/12` — Docker bridge/overlay networks
- `127.0.0.1/8` — localhost

Configurable via the web UI settings page. The Maddy config template uses these CIDRs in its `submission` endpoint ACL. Any IP outside these ranges on port 25 is rejected. Port 587 always requires SMTP AUTH regardless of source IP.

---

## Credential Encryption

Relay SMTP passwords are encrypted at rest in SQLite:
- **Algorithm:** AES-256-GCM (authenticated encryption)
- **Key derivation:** The encryption key is derived from `SIGNPOST_SECRET_KEY` environment variable using HKDF-SHA256 with a fixed info string ("signpost-relay-credentials")
- **Nonce:** Random 12-byte nonce per encryption, stored alongside ciphertext in `password_nonce` column
- **Key rotation:** Changing `SIGNPOST_SECRET_KEY` requires re-encrypting all stored credentials (admin action in web UI)

Backup archives are encrypted with a user-provided password:
- **Algorithm:** AES-256-GCM
- **Key derivation:** Argon2id from user-provided password + random salt
- **The backup password is not stored** — user must remember it. The bcrypt hash in settings is only used to verify the password before creating a backup (UX convenience, not security).

## s6-overlay Process Ordering

1. **s6-rc** starts `signpost-web` first (dependency: none)
2. SignPost Web checks for `/data/signpost/signpost.db` — if missing, runs initial setup (creates DB, generates default `maddy.conf` with self-signed TLS)
3. SignPost Web writes `maddy.conf` and signals readiness via a notification file (`/run/signpost-web-ready`)
4. **s6-rc** starts `maddy` second (dependency: `signpost-web`)
5. Maddy reads the generated `maddy.conf` and begins accepting mail
6. On config changes via web UI: new `maddy.conf` written, then SIGHUP sent to Maddy PID

## Maddy Config Template (Representative Example)

```
# Generated by SignPost — do not edit manually
# Regenerated on config changes via web UI

$(hostname) = {env:SIGNPOST_HOSTNAME}
$(primary_domain) = {env:SIGNPOST_DOMAIN}

tls file /data/signpost/tls/fullchain.pem /data/signpost/tls/privkey.pem

# Inbound SMTP (local services, network trust)
smtp tcp://0.0.0.0:25 {
    hostname $(hostname)
    source {
        # Network trust CIDRs — generated from settings
        {{range .NetworkTrustCIDRs}}
        match_src_ip {{.}}
        {{end}}
    }
    deliver_to &sign_and_relay
}

# Submission (authenticated, STARTTLS)
submission tcp://0.0.0.0:587 {
    hostname $(hostname)
    tls &tls_config
    auth &local_auth
    deliver_to &sign_and_relay
}

# DKIM signing + relay pipeline
pipeline sign_and_relay {
    {{range .Domains}}
    modify.dkim {
        domains {{.Name}}
        selector {{.DKIMSelector}}
        key_path /data/signpost/dkim_keys/{{.Name}}.key
    }
    {{end}}

    # Relay configuration
    {{if .RelayEnabled}}
    deliver_to smtp tcp://{{.RelayHost}}:{{.RelayPort}} {
        starttls yes
        auth plain {{.RelayUsername}} {{.RelayPassword}}
    }
    {{else}}
    deliver_to remote
    {{end}}
}
```

Note: This is a simplified representative example. The actual template will handle per-domain relay configs, multiple domains with different relay methods, and edge cases.

## Error Handling Philosophy

- **Config reload failure:** If Maddy rejects a generated config (SIGHUP returns error), the web app rolls back to the previous known-good config, re-signals Maddy, and shows the error in the UI. Previous config is always preserved as `maddy.conf.bak`.
- **DKIM key generation failure:** Surfaced as an error in the UI. Domain remains in "unconfigured" state. Retry available.
- **ACME/LE failure:** Falls back to existing cert (or self-signed). Error shown on TLS page with troubleshooting guidance. Maddy's built-in ACME has dual-CA fallback (LE prod → LE staging).
- **SQLite corruption:** On startup, SignPost runs `PRAGMA integrity_check`. If corrupt, it attempts recovery from the most recent backup if auto-backup was enabled. Otherwise, surfaces error with instructions.
- **Relay unreachable:** Maddy queues the message and retries per standard SMTP retry logic (configurable). Failed deliveries appear in the mail log.

## Logging Strategy

- **Maddy logs:** Captured by s6-overlay's log service, written to `/data/signpost/logs/maddy/`. Rotated by s6-log (size-based, configurable).
- **SignPost Web logs:** Structured JSON to stdout (captured by s6-log), written to `/data/signpost/logs/web/`.
- **Mail activity log:** Parsed from Maddy's log output by the web app and stored in the `mail_log` SQLite table for the UI's log viewer. Old entries pruned after 30 days (configurable).
- **Docker-level:** Both log streams also available via `docker logs` for users who prefer centralized logging.

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD curl -f http://localhost:8080/api/v1/healthz || exit 1
```

- `503` if Maddy is down or config is invalid

## Upgrade / Migration Strategy

- SQLite schema changes are versioned in `schema_migrations` table
- On startup, SignPost auto-applies any pending migrations (forward-only)
- Maddy config template changes take effect on next config regeneration (automatic on startup)
- Users are advised to backup before upgrading (`signpost backup` CLI or web UI)
- Breaking changes documented in CHANGELOG.md and release notes
- Maddy version pinned to specific release (e.g., `v0.9.2`), upgraded deliberately with testing

## Implementation Phases

### Phase 1 — MVP (v0.1.0)
Core mail relay with DKIM signing, no web UI yet:
- Dockerfile with s6-overlay + Maddy + Go web app skeleton
- SQLite DB with domains and relay_configs tables
- Maddy config template generation from DB
- DKIM key generation and management
- Single domain support (drcs.ca)
- Port 25 with network trust, port 587 with SMTP AUTH
- Self-signed TLS
- CLI: `signpost test` health check
- Basic REST API (domains CRUD, relay config, test send)
- Integration tests with Testcontainers
- GitHub repo setup (CI, Dependabot, .gitignore, README)

### Phase 2 — Web UI (v0.2.0)
- React frontend with Vite + Tailwind + shadcn/ui
- Dashboard, Domains page, Relay Config page
- DNS Helper (shows required records)
- DNS Validator (live DNS checks)
- Setup Wizard (first-run experience)
- Mail log viewer

### Phase 3 — TLS & Security (v0.3.0)
- Let's Encrypt ACME DNS-01 via Cloudflare
- Manual cert upload
- TLS configuration page
- Security audit page
- SMTP user management page

### Phase 4 — Polish & Release (v1.0.0)
- Backup/restore
- Multi-domain support
- Comprehensive documentation
- Release automation (GitHub Actions)
- README, CHANGELOG, setup guides
- Production hardening and full test suite
