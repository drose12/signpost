# Mail Log & Queue Visibility Design

**Date:** 2026-04-06
**Status:** Approved
**Scope:** Real-time mail logging from Maddy, queue visibility, relay queue wrapping

## Problem

SignPost currently only logs emails sent via the "Test Send" UI button. Regular mail flowing through Maddy from apps (port 25/587) is invisible. Additionally, the Gmail relay path (`target.smtp`) is fire-and-forget with no retry logic — if Gmail temporarily rejects, the mail is lost. There is no visibility into Maddy's delivery queue.

## Solution Overview

1. Parse Maddy's structured log output in real-time to capture all mail events
2. Wrap all relay targets in `target.queue` for retry safety
3. Scan queue directories periodically for stuck/retrying messages
4. Enhance the frontend with search, filters, and a queue tab

## 1. Queue Wrapping for Relay Targets

### Current (fire-and-forget)
```
target.smtp relay_0 {
    targets tcp://smtp.gmail.com:587
    ...
}
source drcs.ca {
    deliver_to &relay_0
}
```

### New (queued with retries)
```
target.smtp relay_target_0 {
    targets tcp://smtp.gmail.com:587
    ...
}
target.queue relay_0 {
    target &relay_target_0
    autogenerated_msg_domain {{.PrimaryDomain}}
    bounce {
        default_destination {
            reject 550 5.0.0 "Refusing to send DSN to non-local address"
        }
    }
}
source drcs.ca {
    deliver_to &relay_0   # unchanged — but now it's a queue
}
```

The `source` blocks continue referencing `&relay_0`. The indirection is transparent. Retry schedule is Maddy's default: exponential backoff starting at 15 minutes, up to 20 attempts.

The existing `remote_queue` for direct delivery is unchanged.

### Template Changes

**File:** `templates/maddy.conf.tmpl`

- Rename raw SMTP targets from `relay_{{$i}}` to `relay_target_{{$i}}`
- Add a `target.queue relay_{{$i}}` block wrapping each `relay_target_{{$i}}`
- Add `log stderr_ts` as a global directive for timestamped log output
- `source` blocks unchanged — still reference `&relay_{{$i}}`

## 2. Log Tailer

### New Package: `internal/logtail`

**File:** `internal/logtail/tailer.go`

A goroutine that tails Maddy's log file and parses structured events into the database.

### Maddy Log Format

```
2026-04-06T14:18:55.123Z MODULE: MESSAGE\t{"field1":"value1","field2":"value2"}
```

Tab character separates human-readable prefix from JSON context. Fields are alphabetically sorted.

### Events Captured

| Maddy Log Message | Module | DB Action | Status |
|---|---|---|---|
| `incoming message` | smtp/submission | INSERT new row | accepted |
| `RCPT ok` | smtp/submission | Update recipients | — |
| `accepted` | smtp/submission | Confirm acceptance | accepted |
| `delivered` | queue | UPDATE status | sent |
| `delivery attempt failed` | queue | UPDATE status + error | deferred |
| `not delivered, permanent error` | queue | UPDATE status + error | failed |
| `RCPT error` | smtp/submission | INSERT with error | rejected |
| `authentication failed` | submission/sasl | INSERT security event | rejected |

### Correlation

Events are correlated by `msg_id` (Maddy assigns this on `incoming message`). A single mail_log row is created on `incoming message` and updated as delivery progresses.

### Resilience

- **File not ready:** Tailer retries every 2 seconds until log file exists
- **File truncation:** Detects via file size decrease, resets to beginning
- **Restart:** File position persisted to `/data/signpost/tailer.pos`
- **Deduplication:** In-memory LRU cache (10k msg_ids) prevents re-processing

### Log File Path

Maddy's stderr must be captured to a file for the tailer. This requires adding an s6 log service for the Maddy process. Create `rootfs/etc/s6-overlay/s6-rc.d/maddy-log/run`:

```bash
#!/command/execlineb -P
s6-log -b -- n10 s1000000 /data/signpost/logs/maddy/
```

This writes Maddy's output to `/data/signpost/logs/maddy/current` with rotation (10 files, 1MB each). The Dockerfile must ensure `/data/signpost/logs/maddy/` exists. The tailer watches `/data/signpost/logs/maddy/current`.

## 3. Queue Scanner

### New Package: `internal/queue`

**File:** `internal/queue/scanner.go`

Periodic goroutine (every 30 seconds) that reads Maddy queue `.meta` files.

### Queue Directories

- `/data/signpost/maddy_state/remote_queue/` — direct delivery queue
- `/data/signpost/maddy_state/relay_0/` (and `relay_1/`, etc.) — per-relay queues

**Note:** Queue directory names must be empirically verified against Maddy 0.9.2's actual behavior. As a fallback, the scanner should dynamically discover queue directories by scanning `maddy_state/` for subdirectories containing `.meta` files, rather than hardcoding names. This makes the scanner resilient to naming changes.

### .meta File Format (JSON)

```json
{
  "From": "csb@drcs.ca",
  "To": ["d@drcs.ca"],
  "FailedRcpts": [],
  "TemporaryFailedRcpts": ["d@drcs.ca"],
  "RcptErrs": {"d@drcs.ca": "dial tcp: connection refused"},
  "TriesCount": {"d@drcs.ca": 3},
  "FirstAttempt": "2026-04-06T14:00:00Z",
  "LastAttempt": "2026-04-06T14:45:00Z"
}
```

### Scanner Output

In-memory snapshot exposed via API. No DB writes — queue state is transient.

## 4. Database Schema Changes

### New Migration

```sql
ALTER TABLE mail_log ADD COLUMN msg_id TEXT;
ALTER TABLE mail_log ADD COLUMN source_ip TEXT;
ALTER TABLE mail_log ADD COLUMN source_port TEXT;
ALTER TABLE mail_log ADD COLUMN attempt_count INTEGER DEFAULT 0;
ALTER TABLE mail_log ADD COLUMN direction TEXT DEFAULT 'outbound';

CREATE INDEX idx_mail_log_msg_id ON mail_log(msg_id);
```

Existing columns unchanged. `LogMail()` continues working for test sends.

### New Function: `LogMailEvent()`

Used by the tailer. Uses upsert semantics (`INSERT ... ON CONFLICT(msg_id) DO UPDATE`):
- On `incoming message`: INSERT new row with status "accepted"
- On delivery events: UPDATE existing row matched by `msg_id`
- If a delivery event arrives for an unknown `msg_id` (e.g., from before the tailer started): INSERT a new row with whatever data is available
- Test send entries (via `LogMail()`) will have NULL `msg_id` — these are never touched by the tailer
- Frontend must handle NULL values for `msg_id`, `source_ip`, `source_port` gracefully

### Status Values

- `accepted` — message received, in transit
- `sent` — successfully delivered
- `deferred` — temporary failure, retrying
- `failed` — permanent failure, gave up
- `rejected` — recipient or auth rejected at SMTP level

## 5. API Changes

### Enhanced: `GET /api/v1/logs`

New query parameters:

| Param | Type | Purpose |
|---|---|---|
| `search` | string | LIKE search across from_addr, to_addr, msg_id, error |
| `from` | ISO date | Start of time range |
| `to` | ISO date | End of time range |
| `status` | string | Filter by status (existing, plus new values) |
| `limit` | int | 1-200, default 50 (existing) |
| `offset` | int | 0+, default 0 (existing) |

Response adds new fields per entry: `msg_id`, `source_ip`, `source_port`, `attempt_count`. Backwards compatible.

### New: `GET /api/v1/queue`

```json
{
  "items": [
    {
      "msg_id": "abc123",
      "from": "csb@drcs.ca",
      "recipients": [
        {
          "address": "d@drcs.ca",
          "attempts": 3,
          "last_attempt": "2026-04-06T14:18:55Z",
          "last_error": "dial tcp: connection refused",
          "status": "retrying"
        }
      ],
      "queued_at": "2026-04-06T14:00:00Z",
      "queue_name": "remote_queue"
    }
  ],
  "count": 1
}
```

Read-only. No pagination (queue should be small).

## 6. Frontend

### Mail Log Page Enhancement

**Header area:**
- Search box (debounced 300ms) — searches from/to/msg_id/error
- Status filter dropdown: All, Accepted, Sent, Failed, Deferred, Rejected
- Date range picker (from/to)
- Clear filters button
- Existing "Clear logs" button

**Table columns:** Time | From | To | Status | Relay | DKIM | Details

- Time: relative with absolute tooltip
- Status: colored badges (accepted=blue, sent=green, deferred=yellow, failed=red, rejected=red outline)
- DKIM: check/X icon
- Clicking a row expands inline detail panel: msg_id, source IP, source port, attempt count, full error, relay host

### Queue Tab

Tab bar at top: **Mail Log** | **Queue**

Queue table columns: Queued At | From | To | Attempts | Last Error | Queue

- Auto-refreshes every 30 seconds
- Empty state: green checkmark, "Queue is empty — all messages delivered"
- Non-empty: yellow banner, "N messages waiting for delivery"

## 7. Startup and Lifecycle

### Startup Sequence

1. DB init (existing)
2. Config generation (existing)
3. Start log tailer goroutine
4. Start queue scanner goroutine (30s interval)
5. Start web server (existing)
6. s6 starts Maddy (existing)

### Graceful Shutdown

Context cancellation stops tailer and scanner goroutines.

### Log Tailer File Handling

- Waits for log file to exist (Maddy may not have started yet)
- Detects truncation and resets
- Persists file position to `/data/signpost/tailer.pos`
